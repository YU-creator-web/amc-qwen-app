📋 1. システムの目的（非エンジニア向け）

一言で言うと：「ベテラン社員の頭の中にある、マニュアルに書いていない判断ノウハウを、AIとの音声対話で引き出して、データとして保存・共有するシステム」です。

🤔 なぜこのシステムが必要なのか

業務マニュアルには「手順」は書いてあっても、「なぜそうするのか」「このケースではどう判断するのか」という経験に裏打ちされた判断基準は書かれていないことがほとんどです。 これを「暗黙知」と呼びます。

ベテラン社員が退職・異動すると、その知識は失われてしまいます。このシステムは、AI（音声ボット）がインタビュアーとなって、ベテランとの自然な対話からその暗黙知を自動で抽出・保存します。

💡 このシステムでできること

① 知識の収集： ベテラン社員とAIが音声で対話。「マニュアルにないケースはどう判断しますか？」といった問いかけから経験則を引き出します。

② 知識の整理： 対話内容をAIが自動分析し、「判断基準・適用条件・注意点」の形式に整理して保存します。

③ 知識の共有： 保存した知識を使って、新人・未熟練者に音声でレクチャーしたり、質問に答えたりします。

🎯 2. 3つの使い方（モード）

🎙️ インタビューモード

誰が使う： ベテラン社員（知識を持っている人）

何をするか： AIボットがインタビュアーとなり、業務マニュアルをベースに「マニュアルに書かれていない判断をする場面はありますか？」などの質問を投げかけます。話者はマイクで答えるだけ。対話終了後に「完了」ボタンを押すと自動的に暗黙知の抽出・保存が始まります。

流れ： 業務区分を選ぶ → マニュアルをアップロード（任意） → 名前を入力 → 音声対話 → 「完了」ボタンで終了・自動抽出

深堀りレベル： スライダーで1〜8段階に設定可能。高いほど同一テーマへの深堀り回数が増えます（デフォルト：4回）。

📋 確認・編集モード

誰が使う： 管理者・コンテンツ担当者

何をするか： 蓄積された暗黙知の一覧を確認・編集・削除できます。過去の対話セッションの録音ログを閲覧したり、再度「解析・保存」したりもできます。業務区分でフィルタリング可能。

特徴： カテゴリ未選択でも全データを表示。セッション履歴の削除も可能。

📚 レクチャーモード

誰が使う： 新人・未熟練者（知識を学ぶ人）

何をするか： 蓄積した暗黙知を使ってAIが音声で教えます。「対話モード（自由に質問できる）」と「テスト・問答モード（ボットが問いかける）」を選べます。

流れ： 業務区分を選ぶ → モードを選ぶ → 名前を入力 → レクチャー開始 → 「完了」ボタンで終了

モード対象ユーザー主な機能状態
インタビュー ベテラン社員 音声インタビュー・自動構造化・DB保存 ✅ 完了
確認・編集 管理者 一覧・編集・削除・セッション履歴・再抽出 ✅ 完了
レクチャー 未熟練者 音声対話・テスト問答 ✅ 基本実装完了

🔧 技術的な構成（エンジニア向け）

【ブラウザ（ユーザーの画面）】 localhost:3000
Next.js 16 (React) — TypeScript / Tailwind CSS
・WebAudio API：マイクから音声をキャプチャ (PCM16 / 24kHz)
・ScriptProcessorNode で 4096サンプル単位で base64 エンコード → 送信
・AudioContext + nextAudioTimeRef でチャンク再生をキューイング
・AudioContext は WebSocket 接続時に事前初期化（最初の語句途切れ防止）
・isBotSpeaking 状態で AudioBufferSourceNode の再生完了を追跡
・VAD: 自動モード(server_vad) / 手動モード(push-to-talk)
・スペースキー：音声送信のショートカット（INPUT/TEXTAREA/BUTTON要素では無効）

↕ WebSocket ws://localhost:8000/ws/voice/{session_id}
↕ REST API http://localhost:8000/api/...

【バックエンド（FastAPI）】 localhost:8000
Python + uvicorn
・WebSocket プロキシ：ブラウザ ↔ Azure 間を中継（双方向 asyncio.gather）
・対話ログ保存：conversation.item.input_audio_transcription.completed / response.audio_transcript.done
・システムプロンプト切替：session.mode に応じて抽出用 / レクチャー用を使い分け
・暗黙知構造化：gpt-5.4 に JSON形式で解析依頼 → TacitKnowledge テーブルへ保存
・asyncio.to_thread() で同期AI呼び出しをイベントループから分離
・グローバル例外ハンドラで500エラー時にも CORS ヘッダーを付与
・SQLite (anmokuchi.db) — SQLAlchemy ORM

↕ WSS wss://{resource}.cognitiveservices.azure.com/openai/realtime
↕ HTTPS {resource}.cognitiveservices.azure.com/openai/deployments/.../chat/completions

【Azure OpenAI (East US 2)】
gpt-realtime-1.5 ← 音声リアルタイム対話（Realtimeモデル）
gpt-4o-mini-transcribe ← 音声→テキスト 文字起こし（UI表示用）
gpt-5.4 ← 暗黙知の構造化・STT補正（Chat Completions API）

💬 非エンジニア向け説明

このシステムは3つの部品から成り立っています。

① ブラウザ（見える部分）： ユーザーが操作する画面です。マイクで話すと声を録音してサーバーに送り、AIの返答音声を受け取って再生します。

② バックエンド（橋渡し）： ブラウザとAzureの間を仲介するサーバーです。ローカルPC上で動きます。会話の記録をデータベースに保存する役割も担います。

③ Azure OpenAI（AI本体）： マイクロソフトのクラウド上で動くAIです。音声を理解し、会話を行い、暗黙知を分析します。インターネット接続が必要です。

✅ 6. 実装済み機能の詳細

🎙️ 音声対話エンジン（VoiceChat.tsx + voice.py）

  • ブラウザのマイクから音声を PCM16 (24kHz) でキャプチャし、base64エンコードして WebSocket 経由でバックエンドへ送信
  • バックエンドは受信した音声データを Azure OpenAI Realtime API にプロキシ転送（双方向ストリーミング）
  • AIからの音声返答チャンクを nextAudioTimeRef を使って順番にキューイング再生（重複再生を防止）
  • AudioContext 事前初期化： WebSocket 接続確立時（ws.onopen）に AudioContext を生成・resume しておくことで、最初の発話語句の途切れを防止。以前は最初の音声チャンク到着時に初期化していたため約200msの切り落としが発生していた
  • isBotSpeaking 状態管理： AudioBufferSourceNode の onended イベントと、全チャンクが再生完了したかどうかのカウンタで正確に「ボットが話し中かどうか」を管理。「発話を止める」ボタンの表示/非表示がこの状態に基づく
  • UI にリアルタイムで文字起こしテキストを表示（gpt-4o-mini-transcribe）
  • VADモード切替：
    • 自動モード：2.5秒の無音を検知して自動的に発話終了と判断
    • 手動モード（Push-to-talk）：ボタンを押したタイミングで発話を送信
  • スペースキーショートカット： スペースキーで音声送信の開始/停止が可能。フォーム入力中（INPUT/TEXTAREA/BUTTON）では無効化
  • 録音中の表示：「録音中」→「解析中...」に変更（STT処理中であることを明示）
  • ボット発話後の待機時間： AI返答後に1秒待機してから録音を再開（以前は2秒）
  • セッションIDに紐づいたシステムプロンプトの自動切替（抽出用 / レクチャー用）
  • STT補正： 音声認識テキストを gpt-5.4 でビジネス文脈の誤変換補正してからDBに保存
  • 完了ボタン： VoiceChat に onComplete コールバックを受け取る設計。録音停止・WebSocket切断・親コンポーネントへのコールバック呼び出しを一括実行

📥 インタビューフロー（ExtractionMode.tsx）

  • 業務区分の選択（設定モーダルで追加・削除も可能）
  • 業務マニュアルのアップロード（PDF / DOCX / TXT 対応）→ テキスト抽出してシステムプロンプトに埋め込み
  • 深堀りレベル設定： 1〜8のスライダーで同一テーマへの深堀り質問回数を設定。セッション作成時にバックエンドへ渡し、AIプロンプトに反映
  • 話者名入力 → セッション作成 → 音声対話開始
  • 「完了」ボタンで録音停止 + WebSocket切断 → 自動的にインタビュー開始（1秒後）→ 結果表示
  • エラー発生時のリカバリ画面（「会話に戻る」「新しいセッションを始める」ボタン）
  • ホームボタン： ヘッダーの「Demo App」タイトルクリックで抽出モードのトップに戻る。key をインクリメントしてコンポーネントをリセット

🔧 ヘッダー・ナビゲーション（Header.tsx / page.tsx）

  • タイトルボタン化： ヘッダー左上の「Demo App」をクリックすると抽出モードのトップに戻る（Brain アイコン付き）
  • 設定（歯車）ボタン： ヘッダー右上に歯車アイコンを追加。クリックで設定モーダルを開く
  • 3つのモードタブ（インタビュー・確認・編集・レクチャー）

⚙️ 設定モーダル（SettingsModal.tsx）

  • 業務区分の一覧表示
  • 新規業務区分の追加（Enter キーでも追加可能）
  • 業務区分の削除（削除時に関連するマニュアル・セッション・暗黙知も CASCADE で削除）
  • モーダル外クリックで閉じる
  • 追加・削除後に onCategoryChange コールバックで抽出モードのカテゴリ一覧を自動更新

📋 確認・編集モード（ReviewMode.tsx）

  • 全暗黙知を業務区分フィルタなしで一括表示（カテゴリ選択でフィルタリング可）
  • 各暗黙知のインライン編集（業務フロー名・判断基準・適用条件・備考）
  • 各暗黙知の削除
  • セッション履歴タブ：過去セッションの一覧（話者名・開始日時 JST表示）
  • セッションの発話ログ閲覧（チャット形式）
  • 過去セッションから「解析・保存」ボタン（再抽出・追加保存）
  • セッションの削除（発話ログも CASCADE で一緒に削除）

📚 レクチャーモード（LectureMode.tsx）

  • 対話モード：学習者が自由に質問し、AIが暗黙知DBをもとに回答
  • テスト・問答モード：AIが積極的に問いかけを行う
  • 業務区分・名前を選択してレクチャー開始
  • 音声対話は VoiceChat コンポーネントを再利用
  • 完了コールバック修正済： onComplete={() => setSession(null)} を正しく渡すよう修正（以前は onComplete is not a function エラーが発生）
  • 最初の発話改善： システムプロンプトに「承知しました・了解しました などの確認フレーズを使わない」ルールを追加

🤖 AI プロンプト設計（ai_service.py）

  • 抽出プロンプト（build_extraction_system_prompt）：
    • マニュアルの章立てに沿って順番に質問
    • 深堀りレベル（deep_dive_level）回数の上限を明示
    • 「ない」という回答でも1〜2回角度を変えて再質問するルール
    • スコープ維持ルール：業務以外の話題には応じない
    • 最初のターンルール：確認フレーズ不使用・マニュアル第1章の内容で即開始
  • レクチャープロンプト（build_lecture_system_prompt）：
    • 蓄積暗黙知をコンテキストとして埋め込み
    • 最初のターンルール：確認フレーズ不使用・すぐに具体的な内容へ
  • 暗黙知構造化（structure_tacit_knowledge）：
    • gpt-5.4 (Chat Completions API) で対話ログを JSON 構造化
    • STT誤変換をボット発言の文脈から補正して解釈
    • コードブロック囲みの JSON を正規化して parse

🗄️ 7. データベース設計（SQLite）

SQLite はファイル1つ（backend/anmokuchi.db）でデータを管理します。サーバー起動時に自動生成されるため、事前の設定は不要です。
テーブル名主なカラム説明
business_categories id, name, created_at 業務区分（例：口座開設、銀行口座の不正利用調査）。起動時に「口座開設」が自動挿入される。設定モーダルから追加・削除可能
manuals id, business_category_id, file_name, file_path, content_text, uploaded_at アップロードされた業務マニュアルのメタ情報とテキスト
sessions id, business_category_id, speaker_name, mode, started_at, ended_at 対話セッション。mode は extraction / lecture_test / lecture_dialogue
dialogue_logs id, session_id, speaker(user/bot), text, timestamp 発話ログ。sessions 削除時に CASCADE で連鎖削除
tacit_knowledge id, business_category_id, session_id, business_flow_name, judgment_criteria, applicable_conditions, notes, created_at, updated_at AIが構造化した暗黙知。1セッションから複数件保存される

テーブルの関係

business_categories ─── 1:N ─── manuals
business_categories ─── 1:N ─── sessions ─── 1:N ─── dialogue_logs
business_categories ─── 1:N ─── tacit_knowledge
sessions ─── 1:N ─── tacit_knowledge

☁️ 11. EC2 デプロイについて

基本構成

現在のシステムは、ローカル PC 上で動作するように設計されていますが、AWS EC2 インスタンス上に同じ構成を展開することで、イントラネット・インターネット経由での利用が可能になります。

📦 SQLite のデータ保存場所（EC2）

  • SQLite のデータは backend/anmokuchi.db というファイルに保存されます
  • EC2 では EBS（Elastic Block Store） にマウントされたディスク上に保存されるため、インスタンスを再起動してもデータは消えません
  • ただし、インスタンスを削除すると EBS も削除されます（スナップショット等でバックアップ推奨）

🏢 複数部署への展開（推奨アーキテクチャ）

部署ごとに扱う業務・暗黙知の種類が異なるため、部署ごとに独立したインスタンスを立てる方法が推奨です。
A 複数インスタンス（推奨）
  • 部署ごとに EC2 インスタンスを1つずつ立てる
  • 各インスタンスが独立した SQLite DB を持つ
  • データが完全に分離されプライバシーが守られる
  • 1つのインスタンスが落ちても他に影響しない
  • 同じIPアドレスでも、ポートを変えれば複数サービスを公開可能（例：部署A → :8000/:3000、部署B → :8001/:3001）
B 1インスタンス・業務区分で分離
  • 1つの EC2 インスタンスで複数部署を同居させる
  • 業務区分（business_categories）でデータを論理分離
  • コストは安いが、データが同一DBに混在する
  • 将来的に大量データになったときのパフォーマンス懸念
  • 部署間のデータ分離を UI だけに依存する点がリスク

🌐 同一IPで複数ポートを使う例

# IPアドレス: 136.0.0.5 の場合

部署A（口座開設業務）
Backend: 136.0.0.5:8000 → FastAPI
Frontend: 136.0.0.5:3000 → Next.js
DB: /app_dept_a/backend/anmokuchi.db

部署B（不正利用調査業務）
Backend: 136.0.0.5:8001 → FastAPI
Frontend: 136.0.0.5:3001 → Next.js
DB: /app_dept_b/backend/anmokuchi.db

※ EC2 のセキュリティグループで各ポートのインバウンドルールを開放すること
⚠️ .env ファイルの取り扱い： EC2 に設置する際も .env ファイルにはAzure OpenAI の APIキーが含まれます。gitignore 済みですが、デプロイ時は SCP 等で直接転送し、GitHub には絶対にアップロードしないでください。

🩺 13. トラブルシューティング

症状考えられる原因対処法
バックエンドが起動しない .env ファイルがない、または設定キーが不足している backend/.env を確認。Extra inputs are not permitted エラーの場合は.envに不要なキーがある
Set-ExecutionPolicy が必要と言われる Windows PowerShell のスクリプト実行ポリシー制限 Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser を実行してから再試行
ブラウザに「接続できません」と表示される フロントエンドが起動していない、またはポート3000が使用中 npm run dev が実行中か確認。ポートが占有されている場合は npm run dev -- --port 3001 で別ポートを指定
マイクボタンがグレーアウト（押せない） バックエンドが起動していない、または WebSocket 接続に失敗 http://localhost:8000/health を確認。バックエンドを再起動してから画面をリロード
音声接続はできるが AI が返答しない Azure OpenAI のエンドポイントや APIキーが誤っている .env の AZURE_OPENAI_ENDPOINT と REALTIME_KEY を確認。バックエンドのコンソールにエラーが出ているか確認
AIの最初の発話で語句が途切れる AudioContext が再生時に初期化されていなかった（古いバージョンの問題） 最新コードでは WebSocket 接続時に事前初期化済み。バックエンド・フロントエンドを最新化して確認
「発話を止める」ボタンが消える 古いバージョンの VoiceChat.tsx を使用している 最新コードでは isBotSpeaking 状態で管理済み。フロントエンドを最新化して確認
音声が重なって聞こえる ブラウザのページを複数タブで開いている、または VoiceChat の音声キュー管理に問題 タブを1つだけにする。それでも発生する場合はページをリロード
インタビューで CORS エラーや 500 エラー バックエンドのグローバル例外ハンドラが未設定 最新コードでは @app.exception_handler(Exception) で CORS ヘッダーを付与済み。バックエンドを最新化して再起動
「解析・保存」が 0件 対話内容が少なすぎる、または AI が知識を検出できなかった より具体的な業務判断について話した後で再試行。会話ログが「セッション履歴」に残っていれば再抽出も可能
インタビューでモデルエラー（400） EXTRACT_DEPLOYMENT に推論モデル(gpt-5.4-pro等)が設定されている .env の AZURE_OPENAI_EXTRACT_DEPLOYMENT=gpt-5.4 に修正（非推論モデルを使用）
セッション履歴の時刻がズレる SQLite の UTC タイムスタンプが正しく変換されていない ReviewMode.tsx の formatJST 関数内に「Z」補完の処理があるか確認
npm install でエラーが出る Node.js のバージョンが古い（18未満） node --version で確認。古い場合は Node.js を最新の LTS に更新する
pip install でエラーが出る 仮想環境が有効になっていない、または Python バージョンが古い venv\Scripts\Activate.ps1 を実行してから再試行。プロンプトの先頭に (venv) が表示されていれば有効

6 フロントエンドのビルド
⚠️ メモリ不足対策が必要： t3.micro は RAM 1GB のため、Next.js のビルド中にメモリ不足でプロセスが強制終了（Killed）されます。スワップを追加してからビルドしてください。 
cd ~/amc_app/frontend

# 1. 環境変数を設定（EC2 のパブリックIP を指定）
echo "NEXT_PUBLIC_WS_URL=wss://<EC2のパブリックIP>" > .env.local

# 2. スワップ領域を追加（メモリ不足対策 / 初回のみ）
sudo fallocate -l 1G /swapfile && sudo chmod 600 /swapfile && sudo mkswap /swapfile && sudo swapon /swapfile

⑤ nginx の設定（リバースプロキシ）

7 nginx でフロントとバックを振り分け
ブラウザ
→ https://<IP>/ws/* → nginx → backend:8000 (WebSocket)
→ https://<IP>/api/* → nginx → backend:8000 (REST API)
→ https://<IP>/ → nginx → frontend:3000 
sudo tee /etc/nginx/sites-available/demo-app << 'EOF'
server {
    listen 80;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    ssl_certificate /etc/ssl/certs/demo-app.crt;
    ssl_certificate_key /etc/ssl/private/demo-app.key;

    location /ws/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
    }

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
    }
}
EOF
sudo ln -s /etc/nginx/sites-available/demo-app /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx

⑥ HTTPS（自己署名証明書）

8 自己署名証明書の作成

ブラウザがマイクへのアクセスを許可するには HTTPS が必須です（http:// ではマイク API がブロックされます）。ドメインがない場合は自己署名証明書を使います。

sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/ssl/private/demo-app.key \
  -out /etc/ssl/certs/demo-app.crt \
  -subj "/CN=<EC2のパブリックIP>"

sudo systemctl reload nginx
💡 ブラウザで「保護されていない接続」の警告が表示されます。「詳細設定」→「このサイトにアクセスする」で進んでください。マイクは正常に使えます。

⚠️ ハマりポイントと対処法

問題原因対処法
SSH 接続で Permission denied (publickey) Windows での .pem ファイルのアクセス権限が広すぎる icacls コマンドで権限を自分だけに絞る（手順②参照）
npm run build が途中で Killed になる t3.micro の RAM 1GB では Next.js ビルドがメモリ不足 スワップ 1GB を追加してから再ビルド（手順⑥参照）
設定モーダルで Failed to fetch NEXT_PUBLIC_API_URL が設定されておらず localhost:8000 に接続しようとしている .env.local に NEXT_PUBLIC_API_URL=https://<IP> を追加して再ビルド
https にしたら Failed to fetch が再発 Mixed Content エラー（HTTPS ページから HTTP API は呼べない） NEXT_PUBLIC_API_URL を http:// → https:// に変更、NEXT_PUBLIC_WS_URL を ws:// → wss:// に変更して再ビルド
マイク・スペースキーが反応しない ブラウザは http:// でのマイクアクセスをブロックする 自己署名証明書で HTTPS を設定する（手順⑥参照）
マニュアルをアップロードしてもボットが内容を認識しない pypdf が requirements.txt に含まれておらず、PDF テキスト抽出が無音で失敗する pip install pypdf で手動インストール。既存マニュアルは DB の content_text を手動で再生成する必要あり（requirements.txt への追記も必須）

💬 15. Teams Bot vs 独自UI 音声対話比較

このアプリは当初 Microsoft Teams アプリとして開発することも検討されましたが、独自UIを採用しました。
以下は音声対話UXの観点での比較です。

📊 音声対話 UX 比較表

項目 今のアプリ（独自UI） Teams + Azure Bot Service
音声入力の開始方法 スペースキー長押しで即録音開始 「録音して」などテキスト送信、またはAdaptive Cardのボタン操作が必要
録音中のフィードバック 波形アニメーション・カラーで録音状態を視覚的に表示 ほぼなし（Teamsのチャット画面上に表示できない）
STT（音声→テキスト） Azure STT → 即座にテキスト化・画面に表示 Teams Calling Bot SDK経由が必要（実装が全く別物・非常に複雑）
ボットの返答表示 ストリーミングで文字が流れるようにリアルタイム表示 一括送信（「…」の後にドンと出る）
ボットの読み上げ（TTS） Azure TTSで自動読み上げ・音量調整も可能 Teams標準では非対応・別途実装が必要
会話の継続性 スペースキーを押すだけで即次の発話へ移行できる 毎回テキスト入力またはボタン操作が必要
会話履歴の表示 専用UIでボット・ユーザーを視覚的に分けてデザイン Teamsの標準チャットバブル（カスタマイズに限界あり）
マニュアル参照の表示 専用セクションに整理されて表示 Adaptive Cardで組むが表示の自由度・制約が多い
モード切替 ヘッダーのタブで即座に切替可能 コマンド入力またはカード操作が必要
レスポンス速度 WebSocketで直接通信・低遅延 Bot Service経由で通信経路が増え遅延が発生しやすい
実装難易度 今の構成で完結 Calling Bot SDK・Microsoft Graph API・別途インフラが必要で非常に複雑

🏁 独自UIを選んだ理由

Teams Bot が強い場面：テキストQ&A・通知・ボタン操作など、チャットベースの業務フロー自動化

独自UIが必要な場面：音声での自然な連続対話・リアルタイムフィードバック・カスタムUI要件

このアプリのコアUXである「スペースキー → 録音 → TTS読み上げ → 次の質問」というリズムは、Teams Bot では再現が困難です。
会社のセキュリティ審査の観点とあわせて、独自UIの採用が適切な判断でした。

🏢 16. 会社AWS環境へのデプロイ時の注意点

個人AWSでのフィジビリ検証（セクション14）は完了しました。会社専用のAWS環境にデプロイする際は、社内インフラポリシーにより追加の申請・変更が必要になる可能性があります。

🔴 ほぼ確実に申請が必要なもの

項目 個人環境での対応 会社環境で必要なこと
セキュリティグループ（ポート開放） コンソールから自分で80・443・22番を開放 インフラチームへの申請が必要なことが多い。「HTTPS（443）とWebSocket通信あり」と明示して依頼する
SSL証明書 自己署名証明書（openssl自作）→ブラウザ警告あり 自己署名は企業環境では基本NG。社内CA・AWS ACM・Let's Encrypt のいずれかへの切り替えが必要
ドメイン名 EC2のIPアドレス直打ち（例: https://54.x.x.x） 社内DNSへのドメイン登録申請が必要。IPアドレス直打ちは許可されないことが多い

🟡 環境によっては申請・変更が必要なもの

項目 個人環境での対応 会社環境での想定
リバースプロキシ（nginx） EC2内にnginxを自分でインストール・設定 ALB（AWS Application Load Balancer）が既存の場合、nginxは不要になりALB側のターゲットグループ設定に変わる。WebSocket対応も申請が必要
APIキーの管理（.envファイル） backendフォルダに.envファイルを直接配置 AWS Secrets Manager や Parameter Store への移行を求められる可能性がある。セキュリティポリシー次第
WebSocket通信の許可 nginxのUpgrade/Connectionヘッダーで対応済み ALBやWAF（Web Application Firewall）でWebSocketが明示的に許可されていない場合があり、設定変更申請が必要
マイクアクセス（HTTPS要件） 自己署名証明書でHTTPS化して対応 適切な証明書でHTTPS化されていれば問題なし。HTTP環境では引き続きマイクが使えないため必ずHTTPS必須
アクセス制限 誰でもアクセス可能 社内ネットワーク・VPN経由のみに制限される可能性がある（セキュリティグループのインバウンドを社内IPに限定）

🔄 個人環境から会社環境へのコンポーネント対応表

個人環境（フィジビリ） 会社環境での代替 アプリコードの変更
nginx（SSL終端・振り分け） ALB + ACM（自動証明書更新） 不要（アプリはlocalhost:8000/3000のまま）
自己署名証明書（openssl） 社内CA または ACM 不要
IPアドレス直打ち 社内DNS登録ドメイン .env.local の NEXT_PUBLIC_API_URL と NEXT_PUBLIC_WS_URL をドメインに変更してリビルド必要
.envファイル AWS Secrets Manager（要確認） 必要になる場合あり（backendのsettings読み込み部分の修正）

📋 インフラチームへの依頼時に伝えること

アプリの特性（必ず伝える）：
・ブラウザからマイク（getUserMedia API）を使うため、HTTPS必須
・バックエンドとの通信に WebSocket（wss://） を使用
・フロントエンド（Next.js）とバックエンド（FastAPI）の2プロセス構成
・バックエンドは localhost:8000、フロントエンドは localhost:3000 で動作

依頼内容の例：
「EC2インスタンスにWebアプリをデプロイします。ブラウザからHTTPS（443番ポート）でアクセスできるようにしたいです。WebSocket通信（wss://）も使用します。SSL証明書の発行とドメイン登録をお願いしたいです。」
⚠️ ドメインが変わった場合は frontend/.env.local の NEXT_PUBLIC_API_URL と NEXT_PUBLIC_WS_URL を新しいドメインに更新し、フロントエンドを 必ずリビルド（npm run build）してください。
これらの値はビルド時に埋め込まれるため、再起動だけでは反映されません。
